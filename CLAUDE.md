# 지방세 OX 퀴즈 스터디 — 작업 표준

이 저장소는 지방세 수험 자료(4개 법률)를 OX 퀴즈 학습 앱으로 변환하는 프로젝트다.
사용자가 **"N장 해줘"** 또는 **"<법률명> N장 해줘"**라고 요청하면 아래 표준에 따라 처리한다. 법률명이 없으면 지방세기본법을 뜻한다. 산출물의 톤·구조는 기존 장 파일(`app/quizdata-ch05.js` 등)을 참고한다.
기출 1회분 반영은 별도 절차다 — **"<연도> <시험> 기출 해줘"**(예: "2026 지방직 9급 기출 해줘")는 표준 4를 따른다.

## 저장소 구조

| 경로 | 내용 |
|---|---|
| `app/` | 퀴즈 앱 (정적 사이트: `index.html`, `app.js`, `chapters.js`, `quizdata-*.js`) |
| `materials/<법률명>/chNN/` | 법률·장별 원본 자료 업로드 폴더 (문제·이론정리·요점 3종) |
| `project/`, `chats/` | 초기 디자인 핸드오프 번들 (수정하지 않음) |

- 법률 폴더 4개: `지방세기본법`(9개 장) · `지방세징수법`(3개 장) · `지방세법`(10개 장) · `지방세특례제한법`(2개 장)
- 원본 자료 파일명은 자유형이다 (예: `지방세법 제1장 (문제).txt`). 법률·장 번호와 종류(문제/이론정리/요점)로 식별한다. 안내 문구만 남은 템플릿 파일(`문제.txt`·`이론정리.txt`·`요점.txt`)은 해당 장 처리 시 삭제한다.
- 배포: GitHub Pages가 main 브랜치를 자동 배포한다. 작업은 지정 브랜치에서 하고 PR로 main에 병합한다 (병합은 사용자 승인 후).

## "N장 해줘" 워크플로

1. `materials/<법률명>/chNN/`에서 해당 장의 문제·이론정리·요점 자료를 읽는다 (문제만 있어도 퀴즈 변환은 진행).
2. 아래 3개 표준에 따라 OX 퀴즈 / 이론정리(퀴즈 커버리지 기준 재구성) / 체크리스트를 작성한다.
3. 자기 검증(각 표준의 검증 항목 + 하단 공통 검증)을 통과시킨다.
4. 지정 브랜치에 커밋·푸시하고, 사용자에게 결과를 보고한 뒤 요청 시 PR 병합으로 사이트에 반영한다.

## 표준 1 — 문제 → OX 퀴즈 변환

수험서를 20년간 집필한 출제위원의 관점으로 작성한다.

### 출제 범위와 문항 수
- 자료의 모든 개념·정의·요건·예외를 빠짐없이 문항화한다. 해설에만 나온 내용(표·비교·부연설명)도 별도 문항으로 출제한다.
- 하나의 선지에서 최소 2문항: ① 원문을 거의 그대로 살린 문항 ② 핵심 단어 1~2개를 바꾼 변형 문항 (주체 바꾸기, 있다↔없다, 포함↔제외, 이상↔초과, 숫자·기한 변형 등).
- 대비되는 개념(국세↔지방세, 보통징수↔특별징수, 성립↔확정 등)은 서로 뒤바꾼 교차 문항을 추가한다.
- 여러 문제에서 반복되는 동일 개념은 중복 출제하지 않고 대표 문항 + 변형으로 정리한다.

### 난이도 규칙
- 선지 원문의 문장 구조와 길이를 유지한다. 요약 금지.
- ❌ 문항은 앞 70%는 완전히 옳고 마지막 부분에서만 틀리게 구성한다.
- ⭕ 문항도 같은 길이·문체로 작성해 길이·어투로 정답을 유추할 수 없게 한다.
- 함정 기법을 골고루 사용: 정의 바꿔치기 / 주체 바꿔치기 / 한 글자 함정 / 숫자·기한 변형 / 절대적 표현 삽입 / 예외를 원칙처럼 서술.
- 절대적 표현("모든", "반드시", "예외 없이") 함정은 ❌ 문항의 20%를 넘기지 않는다.

### 정답 분포
- 파트 내 ⭕:❌ 비율을 5:5에 가깝게 유지한다.
- 같은 정답이 4문항 이상 연속되지 않게 배치한다.

### 해설 규칙
- 1행: 정답 판정의 핵심 근거 한 문장.
- ❌ 문항: "무엇이 왜 틀렸는지" + "올바른 내용"을 반드시 둘 다 쓴다.
- ⭕ 문항: 단순히 "옳다"로 끝내지 말고 대비 개념이나 헷갈리기 쉬운 포인트 한 가지를 덧붙인다.
- 3~4줄 이내. "시험에 출제되었다", "Q몇 번 참고" 같은 표현 금지.
- 자료에 없는 조문 번호·숫자·예시를 만들어내지 않는다. 자료가 서로 모순되면 실제 기출(연도·직렬 표기가 있는 문제) 쪽을 기준으로 삼고 사용자에게 보고한다.

### 데이터 형식 (`app/quizdata-chNN.js`의 `DATA`)
```js
{ part: P1, answer: "O"|"X", text: "문항 원문", exp: "해설" }
```
- 파트는 주제별로 나누고 상수(P1~Pn)로 선언한다. 앱은 데이터 순서 그대로 표시한다.

### 자기 검증 (Node 스크립트로 자동 점검)
1. 길이·문체만으로 정답이 보이는 문항이 없는가
2. 파트별 ⭕/❌ 비율과 연속 정답(최대 3) 확인
3. 자료에 없는 내용을 지어낸 문항이 없는가
4. 해설이 정답 근거와 정확히 일치하는가

## 표준 2 — 이론정리 작성 (퀴즈 커버리지 기준 재구성)

> **방식 변경(2026-07): "자료 재배치" → "퀴즈 커버리지 기준 재구성".**
> 예전에는 이론정리 원본을 자르고 옮겨 배치만 했다(퀴즈를 대조하지 않았다). 그 결과 기출 해설이 다루는데 이론정리 원본엔 없거나 얕던 개념(예: 세목별 납세의무자·신용카드 납부·정리보류 정의·취득세 60일)이 이론에서 누락되어, **퀴즈엔 나오는데 이론엔 설명이 없는 "떠 있는 문항"**이 다수 생겼다.
> 이제는 **퀴즈(문항·해설)를 커버리지 목표로 삼아, 이론만 읽어도 그 장 모든 문항의 근거를 이해할 수 있도록 재구성**한다.

### 소스와 원칙
- **소스 = ① 이론정리 원본 + ② 요점 + ③ 그 장의 퀴즈(문항 `text`·해설 `exp`).** 세 가지를 모두 근거로 삼는다.
- **핵심 목표(커버리지):** 퀴즈가 묻는 모든 개념·정의·요건·예외·숫자·기한이 이론에 설명돼 있어야 한다. 이론정리 원본에 없거나 얕게만 있어도 **퀴즈 해설(`exp`)에 근거가 있으면 이론에 추가**한다.
- **허용:** 원문을 쉽게 풀어쓰기, 나열식 문단을 표로 재편, 실례·비유 추가(예: 재산세로 성립→확정→소멸 따라가기), 섹션으로 구조화.
- **금지(변하지 않음):** 소스(이론정리·요점·퀴즈 해설) **어디에도 없는 조문 번호·숫자·세목·예시를 지어내지 않는다.** "새 내용 추가"는 오직 **출처 있는 보강**만 뜻한다. 소스가 서로 모순되면 기출(연도·직렬 표기 문제) 해설을 기준으로 삼고 사용자에게 보고한다.

### 구조 규칙
1. 목표 파트 구조 = 그 장의 퀴즈 파트(PART 1~n)와 **동일하게 고정**. 파트 신설·통합·순서 변경 금지.
2. **각 파트는 `sec`(섹션)로 2~5개로 나눈다.** 섹션 없는 평면 나열 금지(문항 수 대비 이론이 얇아지는 주범). 섹션 제목 번호는 파트 번호 기준(예 PART 4 → `4-1`, `4-2`…).
3. 나열 항목(세목·요건·사유·숫자 목록 등)은 되도록 `table`로 정리한다.
4. 여러 파트에 걸치는 공통 개념(다른 파트가 상세히 다루는 것)은 해당 파트엔 **핵심 요약만** 넣고 `📎 심화·참고`로 "상세는 PART n" 상호참조. 같은 내용을 두 파트에 본문으로 중복 서술하지 않는다.

### 파트 내부 구성 (모든 파트 동일)
① 한 줄 핵심 요약(`lead`, 파트 맨 앞 1개) → ② 섹션별 개념 설명(`p`)·표(`table`)·예시(`note ex`)·비유(`note box`) → ③ 🎯 함정 체크(`note warn`, 파트 끝 필수) → ④ 📎 심화·참고(`note tip`, 있는 경우만)

### 데이터 형식 (`var THEORY` — 파트 순서·개수와 인덱스 일치)
블록 종류: `sec`(섹션 제목) / `sub`(소제목) / `lead`(핵심 요약) / `p`(문단) / `tree`(고정폭 트리) / `table`(head+rows, **행별 열 수 = head 길이**) / `note`(v: `ex`|`box`|`warn`|`tip` 중 하나, `title`+`t` 또는 `title`+`list` 조합)

### 함정 체크·대비 표기 — 렌더러 자동 줄바꿈 (데이터에 줄바꿈 문자 넣지 말 것)
`app/app.js`의 `fmtTrap`이 렌더 시 아래 패턴을 **자동으로 두 줄로** 표시한다:
- `📝 "…지문…" → ❌ 왜 틀렸는지` (화살표가 ❌ 앞) → "지문" 다음 줄에서 ❌ 시작
- `A = 틀린값 ❌ → 올바른값 ⭕ (부연)` (화살표가 ❌ 뒤) → ❌는 틀린값 끝에 붙고 `→` 앞에서 줄바꿈 (`A = 틀린값 ❌` / `→ 올바른값 ⭕`)
- `"…지문…"❌ 설명` (인용부호 바로 뒤 ❌) → 줄바꿈
- `A = … (…) / B = … (…)` (대비, `) /`) → ` / `에서 줄바꿈
- 단, 규칙 요약형 `A = B (C면 ❌)`처럼 **괄호 안 종결 마커**(`❌)`)는 일부러 줄바꿈하지 않는다.

### 자기 검증 (제출 전 필수)
1. **[커버리지] 퀴즈 → 이론 대응** — 그 파트 퀴즈 문항(`text`·`exp`)이 묻는 개념·숫자·요건을 이론이 설명하는지 전수 대조. **근거 없는 퀴즈 포인트 0.** 없던 것은 퀴즈 해설을 근거로 추가.
2. **[정합] 숫자·개수 일치** — 이론과 퀴즈 해설의 숫자·기한·개수(예: 사유 N가지)가 어긋나지 않는가. 어긋나면 기출 기준으로 통일하고 보고.
3. **[구조]** 각 파트: `lead` 1개(맨 앞) + `sec` 2개 이상 + 🎯 함정 체크(`note warn`) 1개 이상. 표 열 수 일치, `note.v` 유효(ex|box|warn|tip).
4. **[중복]** 같은 내용이 두 파트에 본문으로 중복되지 않는가.
5. Node 구조 검증 + Playwright 헤드리스로 전 파트 렌더링(JS 오류 0).

> 참고: 이미 만들어진 장의 이론을 개선할 때는, 파트별로 (a) 현재 이론 블록 (b) 그 파트 퀴즈를 나란히 놓고 위 [커버리지]·[정합]·[구조]를 진단한 뒤 부족분을 보강한다(진단 → 보강). 기존에 잘 채워진 블록은 유지하고 누락·구조만 손댄다.

## 표준 3 — 요점 → 함정 체크 + 최종 암기 체크리스트

### 3-1. 파트별 🎯 함정 체크
- 요점 자료의 시험 함정·실전 예시는 항목별로 해당 파트 이론 끝 **🎯 함정 체크** 박스로 분해 배치한다: `{ k: "note", v: "warn", title: "🎯 함정 체크", list: [...] }`
- list 항목 문체: 실전 지문을 인용할 때는 `📝 "…지문 원문…" → ❌ 왜 틀렸는지` 형식, 값 대비형은 `A = 틀린값 ❌ → 올바른값 ⭕ (부연)` 형식, 규칙 요약일 때는 `A = B (C면 ❌)` 형식.
- **줄바꿈은 데이터에 직접 넣지 않는다.** `app/app.js`의 `fmtTrap` 렌더러가 위 형식들(❌ 앞/뒤 화살표, 인용부호 뒤 ❌, `) /` 대비)을 자동으로 두 줄로 표시한다. 상세 패턴은 표준 2의 "함정 체크·대비 표기 — 렌더러 자동 줄바꿈" 참고. (단 규칙 요약형 `(C면 ❌)`의 괄호 안 종결 마커는 줄바꿈되지 않음)

### 3-2. 최종 암기 체크리스트 (`window.QUIZ_CHECKLIST`)
장 전체 총정리는 파트 이론과 분리해 전용 화면 데이터로 만든다. 파트 목록 상단 ⭐ 카드에서 열리며, 항목을 탭하면 암기 완료 체크(localStorage 저장)되고 진행률이 표시된다 — 이 UI는 앱에 이미 구현되어 있으므로 데이터만 아래 형식으로 작성하면 된다.

```js
window.QUIZ_CHECKLIST = {
  lead: "한 줄 안내문 (항목 수 + 탭해서 체크된다는 안내)",
  groups: [
    { title: "주제명 (PART n~m)",   // 퀴즈 파트와 대응되는 그룹 제목
      items: [
        { key: "개념 키워드",        // 명사구, 짧게 (예: "물납", "소급과세 금지 기준")
          o: "정답 포인트",          // 맞는 내용 한 줄 (예: "재산세만 허용")
          x: "함정 포인트" }         // "…나오면 틀림" 문체로 시험 함정을 명시
      ] }
  ]
};
```

- 항목당 key/o/x 3요소 필수. o는 초록 ⭕, x는 빨강 ❌로 렌더링되므로 **한 항목 = 헷갈리는 두 갈래의 대비**가 되도록 쓴다.
- 그룹은 퀴즈 파트 순서대로 배열하고, 그룹 제목에 해당 PART 번호를 표기한다.
- 항목 수는 요점 자료의 체크리스트를 기준으로 하되 원문 내용을 유지한다 (제1장은 25항목).
- 체크리스트를 파트 이론 안에 중복으로 넣지 않는다.
- 체크 저장 키(`jibangse_checklist_v1`)는 항목 수가 바뀌면 자동 초기화된다.

## 표준 4 — 기출문제 분산 반영 ("&lt;연도&gt; &lt;시험&gt; 기출 해줘")

기출 1회분(예: `2026 지방직 9급`)을 앱에 반영할 때 쓴다. **"N장 해줘"와 달리 별도 챕터를 만들지 않고, 기출 문제를 세목·주제별로 분해해 기존 장에 스며들게 한다.**

### 자료·저장
- 자료 위치: `materials/기출문제/<연도>/<시험>.txt` (또는 사용자가 채팅으로 제공). 문제·정답표를 이 파일에 먼저 저장한다(원본 보존 = 이후 검증 근거).
- 대개 **해설이 없다.** 정답표만 있으므로 O/X 근거는 정답표 + 해당 장 기존 자료(THEORY·기존 exp)로 확정한다.

### 분산 원칙 (핵심)
- 문제 하나가 여러 세목에 걸치면 **선지 단위로 분해**해 각 선지를 그 세목의 장·파트로 보낸다. 결과적으로 한 회분이 10~16개 장을 건드린다.
- 선지 하나 → ① 기출 원문을 거의 그대로 살린 문항 + ② 핵심 단어 1~2개를 바꾼 변형 문항. 대비 개념은 교차 문항 추가(표준 1과 동일).
- **중복 방지:** 기존 DATA에 사실상 동일한 지문이 있으면 스킵, 같은 논점만 있고 기출 원문 문장이 없으면 기출 원문 1개만 추가.

### 출처 뱃지 (`src`)
- **기출 원문 문항에만** `src: "<연도> <시험>"`(예: `"2026 지방직 9급"`)을 부여한다. **변형 문항에는 넣지 않는다**(실제 기출 지문이 아니므로).
- 앱 `app.js`의 `srcBadge`가 문항 본문 아래 오른쪽에 작은 뱃지로 렌더한다(퀴즈·복습 화면, 오답 기록에 유지). `src` 없는 기존 문항은 미표시 — 하위 호환.

### 이론 커버리지 보강 (표준 2와 동일 원칙)
- 새 기출 문항이 묻는 개념·숫자·요건이 그 파트 THEORY에 없으면 보강한다. 이론 원본에 없어도 **기출 정답표를 근거로** 추가할 수 있다(출처 있는 보강만 허용, 창작 금지).
- 자료·기출 어디에도 없는 조문 번호·숫자를 지어내지 않는다. 자료와 기출이 모순되면 기출(연도·직렬 표기) 우선 + 사용자에게 보고한다.

### 삽입·검증
- **원본 파일 무손상 방식:** 데이터 파일 끝의 `window.QUIZ_CHAPTERS[...] = ...` 등록부 앞에 "기출 보강" 즉시실행 코드 블록을 삽입해 해당 파트 안에 문항·이론을 끼워 넣는다(기존 문항·이론 순서 보존, 진행 기록 안전).
- 삽입 후: 파트 내 ⭕/❌ 비율·연속 정답(최대 3) 재확인(위반 시 파트 내 위치 조정), 파트 연속성 유지, 삽입 후 재로드로 문항 수 검증.
- 수정된 모든 장의 `chapters.js` count·README 수치·`ASSET_VER` 갱신. 체크리스트는 기존 유지가 기본(항목 수 변경 시 사용자 체크 기록 초기화되므로 꼭 필요할 때만, 변경 시 보고).
- 해설 문구에 "(연도 시험)" 출처를 쓰지 않는다(뱃지로 표시). "시험에 출제" 류 표현 금지는 동일.
- 대량 작업이므로 장별 (생성 → 소스 대조 적대적 검증) 병렬 워크플로 사용을 권장한다. 마무리는 공통 검증·배포 절차를 따른다.

## 공통 검증·배포 절차

1. `node`로 quizdata.js 로드: 문항 수·파트 수·필수 필드·표 열 수·note 변형 전수 점검.
2. `python3 -m http.server` + Playwright(`playwright-core`, executablePath `/opt/pw-browsers/chromium`)로 실제 렌더링 확인: 홈 → 목차(⭐ 카드) → 각 파트 이론 → 체크리스트 화면, JS 오류 0.
3. `app/README.md`의 문항 수 등 수치 갱신.
4. **`app/index.html`의 스크립트 버전 쿼리(`?v=YYYYMMDD[a-z]`)를 반드시 올린다** — GitHub Pages 캐시(10분) 때문에 이걸 잊으면 사용자에게 옛 버전이 보인다.
5. 커밋 메시지는 한국어로 변경 요지를 명확히. 지정 브랜치에 푸시 후 PR 생성, **사용자 승인(또는 기존 승인 패턴) 하에 병합**.
6. 배포 실패 시: "pages build and deployment" 워크플로 로그를 확인한다. "Deployment failed, try again later"는 GitHub 서비스 장애이므로 rerun_failed_jobs 재시도 + send_later 자동 재점검을 건다.

## 다장(챕터) 구조 — 새 장 추가 절차

앱은 다장 구조다: 홈 = 챕터 목록, 장별 데이터는 지연 로딩, 진행 기록·체크리스트는 챕터별 localStorage 키로 분리되어 있다. 새 장(N장) 추가 시:

1. `app/quizdata-<id>.js` 생성 — 파트 상수 선언 후 `var DATA / THEORY / CHECKLIST`를 정의하고 마지막에 `window.QUIZ_CHAPTERS["<id>"] = { data: DATA, theory: THEORY, checklist: CHECKLIST };`로 등록 (기존 장 파일 참고)
2. `app/chapters.js`의 `CHAPTER_LIST`에 `{ id, num, title, file, count }` 항목 추가 — **count는 실제 문항 수와 일치**시켜야 홈 진행률이 맞다. title은 `"<법률명> 제N장 (주제)"` 형식
3. `app/index.html`의 `ASSET_VER` 갱신 (지연 로딩 파일에도 같은 버전이 적용됨)
4. 앱 코드는 수정 불필요 — 데이터만 추가하면 된다

챕터 id 규칙 (법률별 접두사, localStorage 키 충돌 방지를 위해 한 번 정하면 변경 금지):
- 지방세기본법 `chNN` (기존 유지) / 지방세징수법 `jsNN` / 지방세법 `sbNN` / 지방세특례제한법 `teNN`
- `CHAPTER_LIST` 배열 순서 = 홈 화면 표시 순서. 법률 순서는 기본법 → 징수법 → 지방세법 → 특례제한법.
