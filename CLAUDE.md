# 지방세 OX 퀴즈 스터디 — 작업 표준

이 저장소는 지방세 수험 자료를 OX 퀴즈 학습 앱으로 변환하는 프로젝트다.
사용자가 **"N장 해줘"**라고 요청하면 아래 표준에 따라 처리한다. 제1장은 이 표준으로 완료되어 있으므로, 산출물의 톤·구조는 `app/quizdata.js`의 기존 내용을 참고한다.

## 저장소 구조

| 경로 | 내용 |
|---|---|
| `app/` | 퀴즈 앱 (정적 사이트: `index.html`, `app.js`, `quizdata.js`) |
| `materials/chNN/` | 장별 원본 자료 업로드 폴더 (문제·이론정리·요점 3종) |
| `project/`, `chats/` | 초기 디자인 핸드오프 번들 (수정하지 않음) |

- 원본 자료 파일명은 자유형이다 (예: `지방세기본법 제2장 (문제).txt`). 장 번호와 종류(문제/이론정리/요점)로 식별한다. 안내 문구만 남은 템플릿 파일은 해당 장 처리 시 삭제한다.
- 배포: GitHub Pages가 main 브랜치를 자동 배포한다. 작업은 지정 브랜치에서 하고 PR로 main에 병합한다 (병합은 사용자 승인 후).

## "N장 해줘" 워크플로

1. `materials/`에서 해당 장의 문제·이론정리·요점 자료를 읽는다 (문제만 있어도 퀴즈 변환은 진행).
2. 아래 3개 표준에 따라 OX 퀴즈 / 이론 재배치 / 체크리스트를 작성한다.
3. 자기 검증(각 표준의 검증 항목 + 하단 공통 검증)을 통과시킨다.
4. 지정 브랜치에 커밋·푸시하고, 사용자에게 결과를 보고한 뒤 요청 시 PR 병합으로 사이트에 반영한다.

## 표준 1 — 문제 → OX 퀴즈 변환

수험서를 20년간 집필한 출제위원의 관점으로 작성한다.

### 출제 범위와 문항 수
- 자료의 모든 개념·정의·요건·예외를 빠짐없이 문항화한다. 해설에만 나온 내용(표·비교·부연설명)도 별도 문항으로 출제한다.
- 하나의 선지에서 최소 2문항: ① 원문을 거의 그대로 살린 문항 ② 핵심 단어 1~2개를 바꾼 변형 문항 (주체 바꾸기, 있다↔없다, 포함↔제외, 이상↔초과, 숫자·기한 변형 등).
- 대비되는 개념(국세↔지방세, 보통징수↔특별징수, 성립↔확정 등)은 서로 뒤바꾼 교차 문항을 추가한다.
- 여러 문제에서 반복되는 동일 개념은 중복 출제하지 않고 대표 문항 + 변형으로 정리한다.

### 난이도 규칙
- 선지 원문의 문장 구조와 길이를 유지한다. 요약 금지.
- ❌ 문항은 앞 70%는 완전히 옳고 마지막 부분에서만 틀리게 구성한다.
- ⭕ 문항도 같은 길이·문체로 작성해 길이·어투로 정답을 유추할 수 없게 한다.
- 함정 기법을 골고루 사용: 정의 바꿔치기 / 주체 바꿔치기 / 한 글자 함정 / 숫자·기한 변형 / 절대적 표현 삽입 / 예외를 원칙처럼 서술.
- 절대적 표현("모든", "반드시", "예외 없이") 함정은 ❌ 문항의 20%를 넘기지 않는다.

### 정답 분포
- 파트 내 ⭕:❌ 비율을 5:5에 가깝게 유지한다.
- 같은 정답이 4문항 이상 연속되지 않게 배치한다.

### 해설 규칙
- 1행: 정답 판정의 핵심 근거 한 문장.
- ❌ 문항: "무엇이 왜 틀렸는지" + "올바른 내용"을 반드시 둘 다 쓴다.
- ⭕ 문항: 단순히 "옳다"로 끝내지 말고 대비 개념이나 헷갈리기 쉬운 포인트 한 가지를 덧붙인다.
- 3~4줄 이내. "시험에 출제되었다", "Q몇 번 참고" 같은 표현 금지.
- 자료에 없는 조문 번호·숫자·예시를 만들어내지 않는다. 자료가 서로 모순되면 실제 기출(연도·직렬 표기가 있는 문제) 쪽을 기준으로 삼고 사용자에게 보고한다.

### 데이터 형식 (`app/quizdata.js`)
```js
{ part: P1, answer: "O"|"X", text: "문항 원문", exp: "해설" }
```
- 파트는 주제별로 나누고 상수(P1~Pn)로 선언한다. 앱은 데이터 순서 그대로 표시한다.

### 자기 검증 (Node 스크립트로 자동 점검)
1. 길이·문체만으로 정답이 보이는 문항이 없는가
2. 파트별 ⭕/❌ 비율과 연속 정답(최대 3) 확인
3. 자료에 없는 내용을 지어낸 문항이 없는가
4. 해설이 정답 근거와 정확히 일치하는가

## 표준 2 — 이론정리 → 파트 구조 재배치

내용을 새로 요약하는 것이 아니라 "기존 내용을 자르고 옮겨 배치"하는 작업이다.

### 배치 규칙
1. 목표 구조는 그 장의 퀴즈 파트(PART 1~n)와 동일하게 고정한다. 파트 신설·통합·순서 변경 금지.
2. 자료의 모든 블록(설명·표·예시·비유·주의)을 정확히 한 파트에 배치한다. 버리는 내용 금지.
3. 어느 파트에도 맞지 않는 내용은 삭제하지 말고 가장 관련 있는 파트 끝에 "📎 심화·참고" 박스로 배치한다.
4. 같은 개념이 흩어져 있으면 주된 설명은 한 파트에 모으고 다른 파트에는 한 줄 상호참조만 남긴다.
5. 원문 문장을 임의로 다시 쓰거나 압축하지 않는다. 위치만 옮긴다.

### 파트 내부 구성 (모든 파트 동일)
① 한 줄 핵심 요약(lead) → ② 개념 설명(원문 유지) → ③ 표·비유·예시 → ④ 🎯 함정 체크 → ⑤ 📎 심화·참고(있는 경우만)

### 데이터 형식 (`window.QUIZ_THEORY` — 파트 순서와 인덱스 일치, 파트 수와 동일한 길이)
블록 종류: `sec`(섹션 제목) / `sub`(소제목) / `lead`(핵심 요약) / `p`(문단) / `tree`(고정폭 트리) / `table`(head+rows, 행별 열 수 일치) / `note`(v: ex|box|warn|tip, title·t·list 조합)

### 자기 검증
1. [누락] 원자료 절·항목 → 배치 파트 대응표를 만들어 사용자에게 보고 (배치 안 된 항목 0)
2. [중복] 같은 내용이 두 파트에 본문으로 중복되지 않는가
3. [정합] 파트 제목과 내용이 어긋나지 않는가
4. Node 구조 검증 + Playwright 헤드리스로 전 파트 렌더링 확인

## 표준 3 — 요점 → 함정 체크 + 최종 암기 체크리스트

- 요점 자료의 시험 함정·실전 예시는 항목별로 해당 파트 끝 **🎯 함정 체크** 박스(note v:"warn")로 분해 배치한다.
- 장 전체 암기 체크리스트는 **`window.QUIZ_CHECKLIST`** 로 분리한다 — 파트 목록 상단 ⭐ 카드에서 열리는 전용 화면에 표시된다. 항목을 파트별 그룹(sub + note v:"box" list)으로 나누되 내용은 원문 그대로 유지한다.
- 체크리스트를 파트 이론 안에 중복으로 넣지 않는다.

## 공통 검증·배포 절차

1. `node`로 quizdata.js 로드: 문항 수·파트 수·필수 필드·표 열 수·note 변형 전수 점검.
2. `python3 -m http.server` + Playwright(`playwright-core`, executablePath `/opt/pw-browsers/chromium`)로 실제 렌더링 확인: 홈 → 목차(⭐ 카드) → 각 파트 이론 → 체크리스트 화면, JS 오류 0.
3. `app/README.md`의 문항 수 등 수치 갱신.
4. 커밋 메시지는 한국어로 변경 요지를 명확히. 지정 브랜치에 푸시 후 PR 생성, **사용자 승인(또는 기존 승인 패턴) 하에 병합**.
5. 배포 실패 시: "pages build and deployment" 워크플로 로그를 확인한다. "Deployment failed, try again later"는 GitHub 서비스 장애이므로 rerun_failed_jobs 재시도 + send_later 자동 재점검을 건다.
